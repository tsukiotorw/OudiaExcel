# AI Development Guide

このドキュメントは AI アシスタントが本プロジェクトへ変更を加える際のルールを定義します。

---

# 基本方針

AI は以下を最優先してください。

1. 可読性
2. 保守性
3. 一貫性

短いコードより、理解しやすいコードを優先します。

---

# 開発フロー

以下の順序を守ります。

```
Research
↓
Design
↓
Test
↓
Implementation
↓
Refactor
```

実装を急がず、設計を優先してください。

---

# Python

対象

- Python 3.13

使用する機能

- dataclass
- Enum / StrEnum
- pathlib
- match文
- 型ヒント

Optional は

```
str | None
```

を使用してください。

---

# Parser

Parser は責務を分離してください。

```
Reader
    ↓
Tokenizer
    ↓
SectionBuilder
    ↓
Parser
    ↓
TimeParser
```

各層は一つの責務のみを持ちます。

---

# Domain Model

Parser は Domain Model を生成することだけを担当します。

Parser が表示処理や Excel 出力を行ってはいけません。

---

# テスト

pytest を前提とします。

新しい機能を追加する場合は、

可能な限り先にテストケースを提案してください。

---

# 設計

新しい責務が増える場合は、

既存クラスへ機能を追加するより、

新しいクラス・モジュールへの分離を優先してください。

---

# コメント

Docstring を記述してください。

コメントは理由を書くために使用します。

コードの説明を書くコメントは避けます。

---

# 命名

意味が分かる名前を使用してください。

略称は OudiaSecond の仕様で使用される名称のみ許可します。

---

# コードスタイル

優先順位

1. 読みやすさ
2. 保守性
3. Pythonらしさ

複雑な内包表記は避けてください。

長い関数は分割してください。

---

## staticmethod

`self` または `cls` を使用しないメソッドは、原則として `@staticmethod` を使用してください。

インスタンス状態に依存しない処理は、通常のインスタンスメソッドとして実装しないでください。

---


# 提案

AI は以下を積極的に提案してください。

- 責務分離
- テストケース追加
- リファクタリング
- ドキュメント更新
- ADR が必要な設計判断

実装だけを提案してはいけません。

設計変更が必要な場合は、先に設計書更新を提案してください。

---


# 最重要ルール

実装速度よりも、設計品質を優先してください。

不明な仕様は推測で実装せず、実データの調査または設計の確認を提案してください。
